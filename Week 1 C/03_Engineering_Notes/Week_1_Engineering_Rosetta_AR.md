## 🏛️ حجر رشيد الهندسي: من اللبنات البصرية إلى النص (The Engineering Rosetta Stone)

يعتبر الانتقال من لغة البرمجة الرسومية `Scratch` إلى لغة `C` النصية نقلة جوهرية في بناء التفكير الهندسي. يوضح هذا الجدول المقارن كيف أن المفاهيم التأسيسية ثابتة تماماً (`Concept Constancy`)، بينما تتغير الصياغة النحوية (`Syntax`) لتمنح المبرمج تحكماً مباشراً في موارد النظام.

```text
===================================================================================
                   THE ROSETTA STONE: SCRATCH VS. C PARADIGMS
===================================================================================

  [ Scratch Visual Block ]                   [ C Textual Syntax ]
  ───────────────────────────────────────────────────────────────────────────────
  say [hello, world]                         printf("hello, world\n");
  ask [What's your name?] and wait           string answer = get_string("What's your name? ");
  set [counter] to (0)                       int counter = 0;
  change [counter] by (1)                    counter++;
  repeat (3) [ ... ]                         for (int i = 0; i < 3; i++) { ... }
  forever [ ... ]                            while (true) { ... }
  if < x < y > then [ ... ]                  if (x < y) { ... }
  define meow (n)                            void meow(int n) { ... }

===================================================================================
```

### 🔍 تحليل التكافؤ المعماري:
1.  **دوال الإخراج والإدخال (`Say & Ask vs. printf & get_string`):**
    في سكراتش، تظهر النصوص في فقاعات حوارية. في C، يتم إرسال المخرجات إلى نافذة الطرفية (`Terminal`) عبر `printf`. ولتجنب التحام المخرجات بسطر الأوامر، نستخدم رمز الهروب الصريح `\n` لفرض سطر جديد.
2.  **حلقة الأبدية وإشارات التوقف (`Forever vs. while(true)`):**
    كتلة `forever` الرسومية تقابلها برمجياً حلقة `while (true)`. في سطر الأوامر، إذا دخل النظام في حلقة لا نهائية وفقدت السيطرة، يتم استخدام اختصار لوحة المفاتيح `Ctrl + C` لإرسال إشارة المقاطعة الصريحة `SIGINT` وإجبار البرنامج على إنهاء التنفيذ.

## 📐 محاور التقييم الهندسية الثلاثة (The Three Axes of Code Evaluation)

كتابة كود يعمل ليست سوى البداية. في المعايير الهندسية الصارمة لجامعة هارفارد، يتم تقييم كل تطبيق برمجي على ثلاثة محاور رئيسية:

```text
               ▲ 1. Correctness (check50)
              ╱ ╲
             ╱   ╲
            ╱     ╲
           ╱_______╲
2. Design (design50)  3. Style (style50)
```

### 1. الصحة والموثوقية (`Correctness - check50`)
*   **المفهوم:** هل يعمل البرنامج كما هو متوقع؟ هل يتعامل بشكل صحيح مع المدخلات الصريحة والحرجة؟
*   **الأداة التحليلية:** أداة `check50` هي نظام فحص مؤتمت يقوم بإجراء سلسلة من اختبارات الوحدة (`Unit Tests`) للتحقق من المخرجات الدقيقة مقارنة بالمتطلبات.

### 2. التصميم والكفاءة الهندسية (`Design - design50`)
*   **المفهوم:** هل تم تصميم الكود بكفاءة؟ هل يحتوي على عمليات تكرارية لا داعي لها؟ على سبيل المثال، استبدال الشروط المستقلة المتتالية `if` بكتل `if-else if-else` المتصلة لمنع التحقق غير الضروري، أو إلغاء المتغيرات المؤقتة الزائدة (مثل حساب `x + y` مباشرة داخل `printf` بدلاً من تخزينه في متغير ثالث `z`).
*   **الأداة التحليلية:** أداة `design50` لتقييم كفاءة الخوارزميات وقرارات التجريد.

### 3. الأناقة والاتساق البصري (`Style - style50`)
*   **المفهوم:** هل الكود مريح للعين وقابل للقراءة من قبل المهندسين الآخرين؟ يشمل ذلك المسافات البادئة القياسية (`Indentation`)، ومواضع الأقواس `{}`، واستخدام التعليقات التوضيحية `//`.
*   **الأداة التحليلية:** أداة `style50` التي تعمل كمدقق بصري لضمان اتساق الصياغة.

## 🚨 كوارث الذاكرة وأنواع البيانات (Memory Pitfalls & Hardware Disasters)

توفر لغة C تحكماً هائلاً في الذاكرة، ولكن هذا التحكم يتطلب إدراكاً عميقاً للحدود الفيزيائية للعتاد (`Hardware Limits`).

### 1. طفح الأعداد الصحيحة (`Integer Overflow`)
*   **تشريح الخلل:** المتغير من نوع `int` في لغة C له حجم ثابت محجوز في الذاكرة (غالباً 32 بت). الحد الأقصى لرقم صحيح موجب ذو إشارة هو `2,147,483,647`.
*   **تطبيق `dollars` في المحاضرة:** عند مضاعفة مبلغ بالدولار بشكل مستمر `dollars *= 2`، يتجاوز الرقم سعة التخزين القصوى للبتات المتاحة، مما يؤدي إلى "الطفح" (`Overflow`) وانقلاب القيمة فجأة إلى أرقام سالبة أو أصفار.
*   **الحل الهندسي:** استخدام نوع البيانات `long` ذو الـ 64 بت (ومحدد التنسيق `%li`)، مما يؤجل الوصول إلى الحد الأقصى بشكل كبير.

### 2. الاقتطاع في القسمة الصريحة (`Integer Division Truncation`)
*   **تشريح الخلل:** عند قسمة عددين صحيحين `int` (مثلاً `7 / 2`)، تقوم لغة C بعملية قسمة صحيحة بحتة، حيث يتم التخلص تماماً من أي كسور عشرية (`Truncation`)، ليصبح الناتج `3` وليس `3.5`.
*   **التحويل الصريح (`Casting`):** لحل هذه المعضلة، يتم تحويل أحد المتغيرين قسرياً إلى نوع عشري قبل إتمام العملية: `(float) x / y`.

### 3. القصور الكسري (`Floating-Point Imprecision`)
*   **تشريح الخلل:** أجهزة الكمبيوتر تعمل بنظام ثنائي (أصفار وآحاد). بعض الكسور العشرية (مثل `1/3` أو `0.1`) لا يمكن تمثيلها بشكل دقيق تماماً داخل مساحة ذاكرة محدودة.
*   **تجربة `%.50f`:** عند محاولة طباعة عدد عشري حتى 50 خانة عشرية، تظهر أرقام غريبة وغير متوقعة في النهايات. هذا ليس عطلاً، بل هو تجسيد لقصور الدقة في تمثيل الأرقام العشرية بالبتات.

## ⚙️ صيانة النطاق ونماذج الدوال (Variable Scope & Function Architecture)

في لغة C، تحكم قواعد صارمة نطاق رؤية المتغيرات (`Variable Scope`). المتغير المعرف داخل دالة `main` لا يمكن رؤيته داخل دالة أخرى مثل `meow`.

```c
// Demonstrates pass-by-value and variable scope
#include <stdio.h>

void meow(int n); // Prototype

int main(void)
{
    int n = 3; // Variable n exists ONLY within main's scope
    meow(n);   // A COPY of n's value is passed to meow
}

void meow(int n)
{
    // This 'n' is a separate copy in memory
    for (int i = 0; i < n; i++)
    {
        printf("meow\n");
    }
}
```

*   **التمرير بالقيم (`Pass by Value`):** عندما نمرر المتغير `n` من `main` إلى `meow`، لا يتم إعطاء الدالة حق الوصول إلى المتغير الأصلي، بل يتم إنشاء "نسخة مستقلة" من القيمة داخل ذاكرة الدالة الجديدة.

## 🛠️ أدوات صندوق سطر الأوامر (The Linux CLI Essential Toolbox)

تعد واجهة سطر الأوامر (`CLI`) الأداة الأكثر احترافية وسرعة لإدارة النظام مقارنة بالواجهات الرسومية.

| الأمر الصريح (`CLI Command`) | الوظيفة التشغيلية في نواة النظام |
| :--- | :--- |
| `ls` | عرض محتويات المجلد الحالي (`List directory contents`) |
| `cd` | تغيير مجلد العمل الحالي (`Change directory`) |
| `mkdir` | إنشاء مجلد جديد (`Make directory`) |
| `cp` | نسخ الملفات أو المجلدات (`Copy`) |
| `mv` | نقل أو إعادة تسمية الملفات (`Move / Rename`) |
| `rm` | حذف الملفات نهائياً (`Remove files`) |
| `rmdir` | حذف المجلدات الفارغة (`Remove directory`) |
