## 🏛️ تشريح مراحل المترجم الأربع (The 4 Stages of Clang Compilation)

عندما يكتب الطالب أمر `make hello` في سطر الأوامر، يبدو وكأن النظام يحول ملف `hello.c` إلى برنامج تشغيلي `hello` في خطوة سحرية واحدة. لكن في هندسة النظم العميقة، يقوم محرك `make` باستدعاء مترجم `Clang` (أو `GCC`) ليمر الكود بأربع مراحل معمارية منفصلة وصارمة.

```text
===================================================================================
                       THE 4 STAGES OF COMPILATION (clang)
===================================================================================

  [ Source Code: hello.c ]
            │
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 1. PREPROCESSING (المعالجة المسبقة)                    │ ──> Expands #include & macros
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.i - Expanded Source)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 2. COMPILING (الترجمة إلى لغة التجميع)                  │ ──> Generates Assembly
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.s - Assembly Code)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 3. ASSEMBLING (التجميع إلى كود ثنائي)                   │ ──> Generates Object Code (0s & 1s)
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.o - Raw Object File)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 4. LINKING (الربط مع المكتبات القياسية)                │ ──> Merges stdio.o & cs50.o
  └────────────────────────────────────────────────────────┘
            │
            ▼
  [ Final Standalone Binary: ./hello ]

===================================================================================
```

## 🔬 الغوص العميق في كل مرحلة تشغيلية

### 1. مرحلة المعالجة المسبقة (`Preprocessing`)
*   **المهمة التشغيلية:** يبحث المعالج المسبق عن أي سطر يبدأ برمز المربع `#` (مثل `#include <stdio.h>` أو `#include <cs50.h>`). 
*   **التشريح الداخلي:** يقوم النظام حرفياً بفتح ملف الرأس القياسي `stdio.h` الموجود في مكتبات النظام، ونسخ محتوياته (التي تحتوي على تعريف دالة `printf` ونماذج الدوال الأخرى)، ثم لصقها مباشرة أعلى ملف `hello.c`. كما يقوم في هذه المرحلة بحذف كافة التعليقات البرمجية `//` لأن المعالج لا يحتاجها.

### 2. مرحلة الترجمة (`Compiling`)
*   **المهمة التشغيلية:** تحويل كود لغة C (الذي أصبح الآن ملفاً ضخماً يحتوي على كل المكتبات المضمنة) إلى لغة التجميع (`Assembly Language`).
*   **التشريح الداخلي:** لغة C لا تزال عالية المستوى بالنسبة للمعالج. يقوم المترجم بتحويل الجمل مثل `int z = x + y;` إلى تعليمات معالج منخفضة المستوى (`MOV, ADD, PUSH, POP, CALL`) مخصصة لهندسة المعالج المستهدف (مثل `x86_64` أو `ARM64`).

### 3. مرحلة التجميع (`Assembling`)
*   **المهمة التشغيلية:** تحويل تعليمات الأسمبلي النصية إلى تعليمات ثنائية (`Machine Code` / `Object Code`) تتكون بالكامل من الأصفار والآحاد.
*   **التشريح الداخلي:** يقوم برنامج المجمع (`Assembler`) بترجمة كل أمر أسمبلي إلى شفرة تشغيلية (`Opcode`). الناتج هو ملف يسمى `hello.o`. هذا الملف ثنائي ويحتوي على الكود الخاص ببرنامجنا فقط، لكنه **لا يستطيع العمل بمفرده بعد** لأنه يحتوي على إشارة لدالة `printf` دون معرفة شفرتها التنفيذية الفعلية.

### 4. مرحلة الربط (`Linking`)
*   **المهمة التشغيلية:** دمج ملف الكود الثنائي `hello.o` مع ملفات الكود الثنائي للمكتبات الخارجية (مثل `cs50.o` و `libc.a / stdio.o`).
*   **التشريح الداخلي:** يقوم الرابط (`Linker`) بدمج كل هذه الملفات الثنائية في جدول عناوين موحد (`Symbol Table`). يقوم باستبدال الإشارات المؤقتة لدالة `printf` و `get_string` بالعناوين الذاكرية الحقيقية لشفرتها الثنائية في المكتبة المجمعة. الناتج النهائي هو الملف التشغيلي الواحد `hello`.

## ⚙️ التمثيل العتادي للبيانات في الذاكرة (Hardware Data Representation)

لفهم أسباب المشاكل الهندسية مثل طفح الأعداد (`Overflow`) والقصور الكسري (`Imprecision`)، يجب فحص معمارية الذاكرة على المستوى البتي (`Bit-Level`).

```text
┌──────────────────────────────────────────────────────────────┐
│  32-Bit Signed Integer Architecture (Two's Complement)       │
│  [S] [ 31 Bits for Magnitude / Two's Complement ]            │
│   ▲                                                          │
│   └── Sign Bit (0 = Positive, 1 = Negative)                  │
└──────────────────────────────────────────────────────────────┘
```

### 1. معمارية الأعداد الصحيحة والمكمل الثنائي (`Two's Complement`)
في لغة C، يتم تخزين الأعداد الصحيحة `int` في 32 بت. تم تخصيص البت الأخير (الأكثر أهمية `MSB`) كـ **بت الإشارة (`Sign Bit`)**:
*   إذا كان البت `0`، فالرقم موجب.
*   إذا كان البت `1`، فالرقم سالب.
*   **لماذا يحدث الـ Overflow؟** عندما نصل إلى الرقم `2,147,483,647`، تكون كافة الـ 31 بت المخصصة للرقم مشغولة بالآحاد `111...1` وبت الإشارة `0`. إذا أضفنا `1`، يحدث ترحيل بتي (`Carry`) يغير بت الإشارة إلى `1`، مما يخدع المعالج ليعتبر أن الرقم تحول فجأة إلى أقصى رقم سالب `-2,147,483,648`.

### 2. معيار IEEE 754 للأعداد العشرية (`IEEE 754 Floating-Point Standard`)
تخزين الأعداد العشرية `float` (4 بايت / 32 بت) يعتمد على تقسيم البتات إلى 3 مكونات أساسية:
```text
┌────────┬──────────────────┬──────────────────────────────────┐
│ Sign   │ Exponent (8 bit) │ Mantissa / Fraction (23 bits)    │
│ (1 bit)│                  │                                  │
└────────┴──────────────────┴──────────────────────────────────┘
```
*   **بت الإشارة (`Sign`):** بت واحد (0 موجب، 1 سالب).
*   **الأس (`Exponent`):** 8 بت تمثل حجم الرقم (مضروباً في قوى الأساس 2).
*   **الكسر أو المعامل (`Mantissa / Fraction`):** 23 بت تمثل الدقة الكسرية للرقم.
*   **تشريح القصور الكسري (`Imprecision`):** نظراً لأننا نملك 23 بت فقط لتمثيل الكسر، فإن الأرقام غير المنتهية في الأساس الثنائي (مثل `0.1` أو `0.2`) يتم اقتطاعها وتقريبها لأقرب رقم بتي متاح. عند استخدام أمر `printf("%.50f\n", x/y);`، يكشف النظام هذا الاقتطاع الداخلي للبتات.

## 🧱 إدارة الذاكرة ونطاقات إطار التكدس (Memory Stack Frames & Scope)

عند استدعاء الدوال (مثل استدعاء `meow` من `main`)، يتعامل المعالج مع الذاكرة بنظام **تكدس الاستدعاء (`Call Stack`)**.

```text
===================================================================================
                       THE MEMORY CALL STACK ARCHITECTURE
===================================================================================

  ┌────────────────────────────────────────────────────────┐
  │  [ meow() Stack Frame ]                                │ ──> Contains local copy of 'n'
  ├────────────────────────────────────────────────────────┤
  │  [ main() Stack Frame ]                                │ ──> Contains original 'n = 3'
  ├────────────────────────────────────────────────────────┤
  │  [ Available Free Heap / Memory ]                      │
  └────────────────────────────────────────────────────────┘

===================================================================================
```

*   **العزلة الذاكرية (`Memory Isolation`):** كل دالة تحصل على إطار تكدس خاص بها (`Stack Frame`). عندما نرسل المتغير `n` من `main` إلى `meow`، يقوم المعالج بنسخ القيمة من إطار `main` ووضعها في إطار `meow`. بمجرد انتهاء عمل دالة `meow`، يتم تدمير إطارها بالكامل، وتبقى قيم `main` الأصلية محمية ومعزولة دون أي تغيير.
