🏛️ تشريح مراحل المترجم الأربع (The 4 Stages of Clang Compilation)

عندما يكتب الطالب أمر make hello في سطر الأوامر، يبدو وكأن النظام يحول ملف hello.c إلى برنامج تشغيلي hello في خطوة سحرية واحدة. لكن في هندسة النظم العميقة، يقوم محرك make باستدعاء مترجم Clang (أو GCC) ليمر الكود بأربع مراحل معمارية منفصلة وصارمة.

===================================================================================
                       THE 4 STAGES OF COMPILATION (clang)
===================================================================================

  [ Source Code: hello.c ]
            │
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 1. PREPROCESSING (المعالجة المسبقة)                    │ ──> Expands #include & macros
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.i - Expanded Source)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 2. COMPILING (الترجمة إلى لغة التجميع)                  │ ──> Generates Assembly
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.s - Assembly Code)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 3. ASSEMBLING (التجميع إلى كود ثنائي)                   │ ──> Generates Object Code (0s & 1s)
  └────────────────────────────────────────────────────────┘
            │  (Produces hello.o - Raw Object File)
            ▼
  ┌────────────────────────────────────────────────────────┐
  │ 4. LINKING (الربط مع المكتبات القياسية)                │ ──> Merges stdio.o & cs50.o
  └────────────────────────────────────────────────────────┘
            │
            ▼
  [ Final Standalone Binary: ./hello ]

===================================================================================

🔬 الغوص العميق في كل مرحلة تشغيلية

1. مرحلة المعالجة المسبقة (Preprocessing)

يبحث المعالج المسبق عن أي سطر يبدأ برمز المربع # (مثل #include <stdio.h> أو #include <cs50.h>). يقوم النظام حرفياً بفتح ملف الرأس القياسي stdio.h الموجود في مكتبات النظام، ونسخ محتوياته (التي تحتوي على تعريف دالة printf ونماذج الدوال الأخرى)، ثم لصقها مباشرة أعلى ملف hello.c. كما يقوم في هذه المرحلة بحذف كافة التعليقات البرمجية // لأن المعالج لا يحتاجها.

2. مرحلة الترجمة (Compiling)

تحويل كود لغة C (الذي أصبح الآن ملفاً ضخماً يحتوي على كل المكتبات المضمنة) إلى لغة التجميع (Assembly Language). لغة C لا تزال عالية المستوى بالنسبة للمعالج. يقوم المترجم بتحويل الجمل مثل int z = x + y; إلى تعليمات معالج منخفضة المستوى (MOV, ADD, PUSH, POP, CALL) مخصصة لهندسة المعالج المستهدف (مثل x86_64 أو ARM64).

3. مرحلة التجميع (Assembling)

تحويل تعليمات الأسمبلي النصية إلى تعليمات ثنائية (Machine Code / Object Code) تتكون بالكامل من الأصفار والآحاد. يقوم برنامج المجمع (Assembler) بترجمة كل أمر أسمبلي إلى شفرة تشغيلية (Opcode). الناتج هو ملف يسمى hello.o. هذا الملف ثنائي ويحتوي على الكود الخاص ببرنامجنا فقط، لكنه لا يستطيع العمل بمفرده بعد لأنه يحتوي على إشارة لدالة printf دون معرفة شفرتها التنفيذية الفعلية.

4. مرحلة الربط (Linking)

دمج ملف الكود الثنائي hello.o مع ملفات الكود الثنائي للمكتبات الخارجية (مثل cs50.o و libc.a / stdio.o). يقوم الرابط (Linker) بدمج كل هذه الملفات الثنائية في جدول عناوين موحد (Symbol Table). يقوم باستبدال الإشارات المؤقتة لدالة printf و get_string بالعناوين الذاكرية الحقيقية لشفرتها الثنائية في المكتبة المجمعة. الناتج النهائي هو الملف التشغيلي الواحد hello.

⚙️ التمثيل العتادي للبيانات في الذاكرة (Hardware Data Representation)

لفهم أسباب المشاكل الهندسية مثل طفح الأعداد (Overflow) والقصور الكسري (Imprecision)، يجب فحص معمارية الذاكرة على المستوى البتي (Bit-Level).

┌──────────────────────────────────────────────────────────────┐
│  32-Bit Signed Integer Architecture (Two's Complement)       │
│  [S] [ 31 Bits for Magnitude / Two's Complement ]            │
│   ▲                                                          │
│   └── Sign Bit (0 = Positive, 1 = Negative)                  │
└──────────────────────────────────────────────────────────────┘

💡 معمارية الأعداد الصحيحة والمكمل الثنائي (Two's Complement)

في لغة C، يتم تخزين الأعداد الصحيحة int في 32 بت. تم تخصيص البت الأخير (الأكثر أهمية MSB) كـ بت الإشارة (Sign Bit): إذا كان البت 0، فالرقم موجب. وإذا كان البت 1، فالرقم سالب.

لماذا يحدث الـ Overflow؟ عندما نصل إلى الرقم 2,147,483,647، تكون كافة الـ 31 بت المخصصة للرقم مشغولة بالآحاد 111...1 وبت الإشارة 0. إذا أضفنا 1، يحدث ترحيل بتي (Carry) يغير بت الإشارة إلى 1، مما يخدع المعالج ليعتبر أن الرقم تحول فجأة إلى أقصى رقم سالب -2,147,483,648.

1. معيار IEEE 754 للأعداد العشرية (IEEE 754 Floating-Point Standard)

تخزين الأعداد العشرية float (4 بايت / 32 بت) يعتمد على تقسيم البتات إلى 3 مكونات أساسية:

┌────────┬──────────────────┬──────────────────────────────────┐
│ Sign   │ Exponent (8 bit) │ Mantissa / Fraction (23 bits)    │
│ (1 bit)│                  │                                  │
└────────┴──────────────────┴──────────────────────────────────┘
  • بت الإشارة (Sign): بت واحد (0 موجب، 1 سالب).
  • الأس (Exponent): 8 بت تمثل حجم الرقم (مضروباً في قوى الأساس 2).
  • الكسر أو المعامل (Mantissa / Fraction): 23 بت تمثل الدقة الكسرية للرقم.
  • تشريح القصور الكسري (Imprecision): نظراً لأننا نملك 23 بت فقط لتمثيل الكسر، فإن الأرقام غير المنتهية في الأساس الثنائي (مثل 0.1 أو 0.2) يتم اقتطاعها وتقريبها لأقرب رقم بتي متاح. عند استخدام أمر printf("%.50f\n", x/y);، يكشف النظام هذا الاقتطاع الداخلي للبتات.

🧱 إدارة الذاكرة ونطاقات إطار التكدس (Memory Stack Frames & Scope)

عند استدعاء الدوال (مثل استدعاء meow من main)، يتعامل المعالج مع الذاكرة بنظام تكدس الاستدعاء (Call Stack).

===================================================================================
                       THE MEMORY CALL STACK ARCHITECTURE
===================================================================================

  ┌────────────────────────────────────────────────────────┐
  │  [ meow() Stack Frame ]                                │ ──> Contains local copy of 'n'
  ├────────────────────────────────────────────────────────┤
  │  [ main() Stack Frame ]                                │ ──> Contains original 'n = 3'
  ├────────────────────────────────────────────────────────┤
  │  [ Available Free Heap / Memory ]                      │
  └────────────────────────────────────────────────────────┘

===================================================================================

العزلة الذاكرية (Memory Isolation): كل دالة تحصل على إطار تكدس خاص بها (Stack Frame). عندما نرسل المتغير n من main إلى meow، يقوم المعالج بنسخ القيمة من إطار main ووضعها في إطار meow. بمجرد انتهاء عمل دالة meow، يتم تدمير إطارها بالكامل، وتبقى قيم main الأصلية محمية ومعزولة دون أي تغيير.

🏛️ The 4 Stages of Clang Compilation

When a student executes make hello in the terminal, it appears as though the underlying OS magically morphs hello.c into a binary executable hello in a single bound. However, under the hood of systems engineering, make delegates to a robust C compiler (Clang or GCC), initiating a rigorous 4-stage pipeline.

🔬 Deep Internal Forensics of the Toolchain

1. Preprocessing

The preprocessor scans the source code for any preprocessor directives prefixed by a hash symbol # (such as #include <stdio.h> or #include <cs50.h>). The engine physically accesses the header files residing in system libraries, copies their internal function declarations (printf, get_string), and pastes them directly into the top of hello.c. Concurrently, it strips away all developer comments //, as the underlying translation units do not require human annotations.

2. Compiling

Translates the expanded C source file into low-level Assembly code (Assembly Language). While C provides structural abstractions (if, while, int z = x + y;), the target hardware requires primitive CPU instructions. The compiler translates high-level C logic into processor-specific mnemonics (MOV, ADD, PUSH, POP, CALL) tailored to the explicit target architecture (x86_64 or ARM64).

3. Assembling

Converts textual Assembly instructions into raw machine-level binary code (Object Code), composed entirely of ones and zeros. The assembler translates each Assembly mnemonic into a direct processor opcode. The resulting file is named hello.o. This object file contains the binary instructions for our logic alone; it cannot execute independently yet because it contains external references to printf without knowing its actual physical binary implementation.

4. Linking

Merges the standalone object file hello.o with pre-compiled object files of external libraries (cs50.o, libc.a / stdio.o). The linker resolves all external symbol references across multiple object files, mapping them into a unified, executable symbol table (Symbol Table). It replaces placeholder calls for printf with the precise memory addresses of the pre-compiled binary code in the C standard library. The ultimate output is the single, executable binary hello.

⚙️ Hardware Data Representation & Bit-Level Storage

To truly grasp systems phenomena such as integer overflow (Overflow) and floating-point imprecision (Imprecision), one must inspect memory architecture at the fundamental bit level.

│  32-Bit Signed Integer Architecture (Two's Complement)       │
│  [S] [ 31 Bits for Magnitude / Two's Complement ]            │
│   ▲                                                          │
│   └── Sign Code (0 = Positive, 1 = Negative)                 │
└──────────────────────────────────────────────────────────────┘

💡 Signed Integer Architecture & Two's Complement

In C, standard int variables occupy 32 bits of memory. The most significant bit (MSB) is designated as the Sign Bit: If the Sign Bit is 0, the integer is positive. If the Sign Bit is 1, the integer is negative.

The Overflow Mechanism: When an integer reaches its positive maximum (2,147,483,647), all 31 magnitude bits are populated with ones 111...1, and the sign bit is 0. Adding 1 triggers a cascading binary carry across all bits, flipping the sign bit to 1. The processor immediately interprets this binary sequence as the maximum negative number -2,147,483,648.

1. IEEE 754 Floating-Point Standard

The float data type allocates 32 bits (4 bytes) of memory, structurally partitioning the bits into three dedicated hardware fields:

┌────────┬──────────────────┬──────────────────────────────────┐
│ Sign   │ Exponent (8 bit) │ Mantissa / Fraction (23 bits)    │
│ (1 bit)│                  │                                  │
└────────┴──────────────────┴──────────────────────────────────┘
  • Sign Bit: 1 bit indicating positivity or negativity.
  • Exponent: 8 bits representing the magnitude of the number via powers of base 2.
  • Mantissa / Fraction: 23 bits representing the fractional precision of the value.
  • Anatomy of Imprecision: Because only 23 bits are allocated to represent the fractional component, non-terminating binary fractions (such as 0.1 or 1/3) must be truncated and rounded to the nearest representable bit combination. Executing printf("%.50f\n", x/y); forces the system to expose this inherent hardware truncation.

🧱 Call Stack Frames & Memory Scope Isolation

When a function invokes another function (such as main calling meow), the CPU manages execution memory via the Call Stack (Call Stack).

===================================================================================
                       THE MEMORY CALL STACK ARCHITECTURE
===================================================================================

  ┌────────────────────────────────────────────────────────┐
  │  [ meow() Stack Frame ]                                │ ──> Contains local copy of 'n'
  ├────────────────────────────────────────────────────────┤
  │  [ main() Stack Frame ]                                │ ──> Contains original 'n = 3'
  ├────────────────────────────────────────────────────────┤
  │  [ Available Free Heap / Memory ]                      │
  └────────────────────────────────────────────────────────┘

===================================================================================

Stack Frame Isolation: Every function invocation gets allocated a private chunk of memory called a Stack Frame. When passing n from main to meow, the CPU pushes a duplicate copy of n into meow's stack frame. Upon completion of meow, its stack frame is popped off the stack and destroyed, leaving main's original variables fully isolated and unmodified.
