🏛️ التشريح الجنائي لخوارزميات البحث والفرز (Algorithms Forensics)

في هذا القسم من الهندسة العكسية التطبيقية، لن نكتفي بمجرد فهم منطق الخوارزميات نظرياً، بل سنغوص في أعماق التعقيد الزمني والمكاني (Big O Notation) عبر فحص الكود المجمع (Assembly) ومراقبة سلوك الذاكرة وتكدس الاستدعاءات (Call Stack) أثناء تنفيذ خوارزميات البحث والفرز.

===================================================================================
                       THE ALGORITHMS INSPECTION ENGINE
===================================================================================

  [ Unsorted / Unindexed Memory Structures ]
             │
             ├─► Linear Search  ──> O(n)     ──> Sequential Memory Traversal
             ├─► Binary Search  ──> O(log n) ──> Divide & Conquer Pointer Arithmetic
             ├─► Selection Sort ──> O(n²)    ──> Quadratic Register Swaps
             └─► Merge Sort     ──> O(n log n) ──> Recursive Stack Allocation

🔬 أدوات التفكيك الاستقصائية للخوارزميات

1. تحليل البحث الخطي مقابل الثنائي (Linear vs Binary Search)

عند فحص أكواد لغة التجميع (Assembly) الناتجة عن تجميع خوارزميتي البحث الخطي والثنائي، نلاحظ فارقاً جوهرياً في استخدام مسجلات المعالج وقفزات التعليمات (Branching):

# Linear Search (Loop Traversal)
.L2:
    cmpl  %esi, (%rdi,%rax,4)  # Compare current element with target
    je    .L5                  # Jump if equal (found)
    incq  %rax                 # Increment index
    cmpq  %rdx, %rax           # Check loop bound
    jne   .L2                  # Repeat loop

# Binary Search (Pointer Halving)
.L10:
    leaq  (%rsi,%rdx), %rax    # (low + high)
    shrq  $1, %rax             # Divide by 2 (Bitwise Shift Right)
    cmpl  %ecx, (%rdi,%rax,4)  # Compare middle element
    ...

2. فحص إطارات التكدس والعودية (Call Stack & Recursion)

تتميز خوارزمية فرز الدمج (Merge Sort) باعتمادها على العودية (Recursion). أداة objdump توضح لنا كيف ينمو تكدس النظام (Stack) ديناميكياً مع كل استدعاء ذاتي للدالة، مما يفسر التعقيد المكاني O(n) المطلوب لحفظ إطارات التكدس المؤقتة:

$ objdump -d mergesort
...
100004a10: 55                    pushq %rbp
100004a11: 48 89 e5              movq  %rsp, %rbp
100004a14: 48 83 ec 30           subq  $48, %rsp      # Allocating stack space for sub-arrays
...
100004a3b: e8 d0 ff ff ff        callq _mergesort     # Recursive call (left half)
100004a48: e8 c3 ff ff ff        callq _mergesort     # Recursive call (right half)
100004a55: e8 80 01 00 00        callq _merge         # Merge subroutine
...
  • subq $48, %rsp: تخصيص مساحة في التكدس المحلي لحفظ المؤشرات المؤقتة للمصانع الفرعية.
  • callq _mergesort: الاستدعاء العودي المتكرر الذي ينشئ إطارات متراكمة حتى الوصول إلى الحالة الأساسية (Base Case).

⚙️ الأداء الفعلي وكفاءة الكاش (Cache Efficiency & Memory Hierarchy)

من الناحية الهندسية الصارمة، لا يعتمد أداء الخوارزميات فقط على عدد العمليات الحسابية، بل يتأثر بشدة بطريقة جلب البيانات من الذاكرة الرئيسية (RAM) إلى ذاكرة التخزين المؤقت للمعالج (CPU L1/L2 Cache).

💡 تشريح نقل البيانات في المعالج

عقدة القفزات العشوائية (Cache Misses): في الخوارزميات التي تقفز عبر الذاكرة عشوائياً، يضطر المعالج لتفريغ الكاش وإعادة جلب خطوط جديدة من الذاكرة البطيئة (Cache Miss). بينما في المصفوفات الخطية المتجاورة، يقوم المعالج بالجلب المسبق (Prefetching) مما يرفع كفاءة التنفيذ بشكل هائل.

🔬 The Analytical Disassembly Suite

1. Linear vs Binary Search Architectural Traversal

Examining the generated x86_64 Assembly execution paths reveals the stark operational contrast between sequential looping and logarithmic pointer manipulation:

# Linear Search (Loop Traversal)
.L2:
    cmpl  %esi, (%rdi,%rax,4)  # Compare current element with target
    je    .L5                  # Jump if equal (found)
    incq  %rax                 # Increment index
    cmpq  %rdx, %rax           # Check loop bound
    jne   .L2                  # Repeat loop

# Binary Search (Pointer Halving)
.L10:
    leaq  (%rsi,%rdx), %rax    # (low + high)
    shrq  $1, %rax             # Divide by 2 (Bitwise Shift Right)
    cmpl  %ecx, (%rdi,%rax,4)  # Compare middle element
    ...

2. Call Stack Disassembly & Recursive Allocation

Merge Sort fundamentally relies on recursive call execution. Inspecting the machine code via objdump demonstrates exactly how runtime stack frames are allocated dynamically to support the divide-and-conquer paradigm:

$ objdump -d mergesort
...
100004a10: 55                    pushq %rbp
100004a11: 48 89 e5              movq  %rsp, %rbp
100004a14: 48 83 ec 30           subq  $48, %rsp      # Allocating stack space for sub-arrays
...
100004a3b: e8 d0 ff ff ff        callq _mergesort     # Recursive call (left half)
100004a48: e8 c3 ff ff ff        callq _mergesort     # Recursive call (right half)
100004a55: e8 80 01 00 00        callq _merge         # Merge subroutine
...
  • subq $48, %rsp: Explicitly reserves 48 bytes within the current stack frame to maintain sub-array pointer addresses.
  • callq _mergesort: Invokes the recursive self-call, systematically pushing return addresses onto the physical call stack until reaching the base case.

⚙️ Physical Cache Efficiency & Memory Hierarchy

In production-grade engineering, algorithmic execution speed is dictated not solely by asymptotic instruction counts, but heavily by hardware-level memory access patterns across the CPU L1/L2 Cache subsystem.

💡 Deconstructing Hardware Prefetching

The Cache Miss Bottleneck: Algorithms exhibiting non-contiguous, random memory leaps trigger expensive CPU cache misses, forcing processor pipeline stalls while fetching cache lines from slow main memory (RAM). Conversely, linear contiguous array traversals fully benefit from hardware prefetching mechanisms.
