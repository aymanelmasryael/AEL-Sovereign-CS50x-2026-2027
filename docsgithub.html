Chapter 7: Demystifying & Resolving Merge Conflicts

Antigravity Architectural Commentary
Merge conflicts are not system failures; they are a highly logical protective mechanism. When two diverging environments modify the exact same line of code concurrently (e.g., setting `int x = 17;` locally while the cloud contains `int x = 51;`), Git halts the automated merge to prevent silent data corruption. It injects explicit conflict markers (`<<<<<<< HEAD`, `=======`, `>>>>>>>`) and demands that the system architect manually refactor the block and commit the definitive resolution.
<<<<<<< HEAD
int x = 17; // Your local unmerged modification
=======
int x = 51; // Incoming remote modification from GitHub
>>>>>>> 4761626a8d9...
zsh - conflict_resolution
# 1. View unmerged files experiencing active conflicts $ git status # 2. After deleting conflict markers and setting the line to int x = 34;, commit the clean state $ git commit -am "🔧 [Bugfix] Resolve merge conflict in swap.c by setting x to 34" # 3. Push the resolved state up to the cloud server $ git push

Chapter 9: Isolated Sandboxes & Branching (`git branch`, `checkout`, `merge`)

Antigravity Architectural Commentary
Branching enables the creation of fully isolated parallel dimensions within a single repository. System architects can safely construct experimental features (such as encapsulating logic into a dedicated `swap()` function) in a separate branch without threatening the stability of `master`. When switching branches via `git checkout`, the host operating system instantly restructures the live filesystem to match the target branch topology.
zsh - branching
# 1. List active local branches (current branch is highlighted in green with a star) $ git branch # 2. Create a new feature branch (e.g. function) $ git branch function # 3. Switch active working tree to the newly created branch $ git checkout function # 4. Elite fast-track command: Create and checkout a new branch simultaneously $ git checkout -b feature-swap-function # 5. Switch back to the primary master branch $ git checkout master # 6. Merge changes from the experimental branch into the active master branch $ git merge function # 7. Delete the feature branch once its changes are cleanly merged $ git branch -D function
