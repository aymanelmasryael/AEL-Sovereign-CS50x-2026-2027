مقدمة معمارية: القوى الأربع الكبرى لنظام Git (The 4 Superpowers)

في ندوة CS50 التاريخية "It's going to be about Git and GitHub"، يوضح المهندس برايان يو (Brian Yu) أن نظام Git ليس مجرد أداة لحفظ الملفات، بل هو نظام متقدم للتحكم في الإصدارات (Version Control) يمنح مهندس النظم أربع قوى خارقة:

1. تتبع السجل التاريخي

تسجيل دقيق لكل تعديل طرأ على الكود (من أضافه، متى، ولماذا)، مما يجعل اكتشاف الأخطاء وتتبع أسبابها (Debugging) أمراً حتمياً وموثوقاً.

2. المزامنة والتعاون

دمج جهود عدة مهندسين يعملون في ذات الوقت على نفس الملفات عبر مستودع سحابي (GitHub)، لضمان امتلاك الجميع لأحدث نسخة محدثة.

3. التجربة الحرّة والآمنة

القضاء نهائياً على العادة الفوضوية القديمة بإنشاء نسخ متراكمة مثل project_old.c. يتيح لك Git اختبار أفكار جديدة دون المساس بالكود الأصلي السليم.

4. التراجع الفوري عن الأخطاء

القدرة المطلقة على العودة بالزمن إلى أي إصدار مستقر بضغطة زر واحدة، في حال أدت التعديلات الجديدة إلى كسر النظام أو تعطله.

1. التهيئة، ضبط الهوية، واستنساخ المشاريع (Setup, Init & git clone)

الأوامر التأسيسية لتعريف هويتك السيادية كمهندس نظم، تهيئة المستودعات المحلية، واستنساخ المشاريع السحابية لبيئتك المحلية (مثل CS50 IDE أو جهاز Mac).

zsh - config_setup
# 1. ضبط اسم المالك السيادي عالمياً في جهازك $ git config --global user.name "Ayman Elmasry" # 2. ضبط البريد الإلكتروني الرسمي للمالك $ git config --global user.email "info@aymanelmasry.com" # 3. فحص إعدادات الهوية الحالية للتأكد من صحتها $ git config --list # 4. تهيئة مستودع محلي جديد داخل أي مجلد فارغ (تحويله إلى مستودع يتبعه Git) $ git init # 5. استنساخ مستودع سحابي قائم (git clone): ينشئ نسخة مطابقة للمستودع السحابي في جهازك المحلي $ git clone https://github.com/aymanelmasryael/AEL-Sovereign-CS50x-2026-2027.git

2. الفحص، المراقبة، والمقارنة الذرية (git status, diff & log)

أوامر الرادار البرمجي التي تشرحها ندوة CS50 لمعرفة حالة الكود وتتبع التعديلات وسجلات الحفظ بدقة متناهية قبل الإقدام على أي خطوة.

zsh - inspection
# 1. فحص حالة الكود (git status): يخبرك بالفرع الحالي (مثل master)، والملفات المعدلة، وعدد التوثيقات التي تسبق السحاب (ahead of origin) $ git status # 2. عرض الفروقات البرمجية الدقيقة (git diff): يعرض السطور المضافة باللون الأخضر والمحذوفة بالأحمر (قبل حفظها) $ git diff # 3. عرض سجل الحفظ (git log): يعرض تاريخ وثائق الحفظ، هوية أصحابها، التواريخ، ومعرف التوثيق الفريد (Commit Hash) $ git log # 4. عرض سجل الحفظ بأسلوب معماري ناصع ومختصر (سطر واحد لكل توثيق مع رسم بياني) $ git log --oneline --graph --all # 5. عرض تفاصيل آخر عملية حفظ تمت في المستودع (بما في ذلك التعديلات الفعلية) $ git show HEAD

3. التجهيز والتوثيق السيادي (git add & git commit)

عملية النقل الثنائية المعتمدة: تجهيز الملفات في منصة العرض (Staging) أولاً، ثم ختمها بالسجل الدائم (Commit) مع كتابة عنوان معماري يصف التعديل بدقة.

zsh - commit
# 1. إضافة كافة الملفات والتعديلات الحالية إلى منصة التجهيز (Staging) $ git add . # 2. إضافة ملفات محددة بدقة (مثل ملف swap.c كما في شرح ندوة CS50) $ git add swap.c # 3. إنشاء وثيقة حفظ (Commit) مع عنوان معماري احترافي يصف التعديل بوضوح $ git commit -m "🚀 [v1.0] Sovereign System Architect Initialization" # 4. الأمر الذهبي السريع (git commit -am): يدمج خطوتي الإضافة (add) والحفظ (commit) معاً للملفات المتعقبة مسبقاً $ git commit -am "✨ Actually swap the values in swap.c" # 5. تعديل آخر وثيقة حفظ (في حال أردت تغيير الوصف أو إضافة ملف نسيته دون إنشاء توثيق جديد) $ git commit --amend -m "🚀 [v1.0] Sovereign System Architect Initialization (Updated)"

4. فك تشفير وحل نزاعات الدمج (Demystifying Merge Conflicts)

تعتبر نزاعات الدمج (Merge Conflicts) من أكثر المواضيع التي تُرهق المبرمجين، ولكن ندوة CS50 توضح أنها عملية منطقية وبسيطة جداً. تحدث هذه النزاعات عندما يقوم شخصان (أو بيئتا عمل) بتعديل نفس السطر المصدري في ذات الوقت (مثلاً: قيامك محلياً بجعل int x = 17; بينما النسخة على GitHub تم تعديلها لتصبح int x = 51;). عند محاولة السحب (git pull)، يعجز Git عن اتخاذ القرار، فيطلب منك التدخل الحتمي.

<<<<<<< HEAD
int x = 17; // نسختك المحلية الحالية (Your local change)
=======
int x = 51; // التعديل السحابي القادم من GitHub (Incoming remote change)
>>>>>>> 4761626a8d9...

🛠️ خطوات الحل المعماري الصارم لنزاع الدمج:

  1. افتح الملف المتنازع عليه (مثل swap.c) في محرر الأكواد.
  2. احذف علامات النزاع الزائفة كلياً (<<<<<<< HEAD و ======= و >>>>>>>).
  3. قرر الكود النهائي المطلوب (مثلاً، الاحتفاظ بنسختك، أو نسخة السحاب، أو دمجهما معاً لتصبح int x = 34;).
  4. نفذ أوامر التوثيق والدفع التالية لإغلاق النزاع نهائياً:
zsh - merge_conflict_resolution
# 1. فحص الملفات التي تحتوي على نزاعات دمج غير محلولة (unmerged paths) $ git status # 2. بعد تنظيف الملف واختيار الكود النهائي، قم بتوثيق الحل مباشرة $ git commit -am "🔧 [Bugfix] Resolve merge conflict in swap.c by setting x to 34" # 3. دفع الكود النظيف إلى الخادم السحابي لإنهاء النزاع عالمياً $ git push

5. إدارة الفروع وطلبات السحب (Branching & Pull Requests)

تتيح لك الفروع (Branches) عزل الأكواد التجريبية (مثل إنشاء دالة جديدة swap()) دون المساس بالفرع الرئيسي المستقر (master أو main). وعند التبديل بين الفروع (git checkout)، يقوم نظام التشغيل فوراً بتحديث الملفات أمام عينيك في المجلدات لتطابق الفرع المحدد!

zsh - branching_pr
# 1. عرض قائمة الفروع المحلية (الفرع الأخضر المميز بنجمة هو الفرع الحالي) $ git branch # 2. إنشاء فرع تجريبي جديد والانتقال إليه فوراً (الأسلوب المعماري لطلبات السحب) $ git checkout -b feature-swap-function # 3. الانتقال السلس بين الفروع المتاحة (العودة للفرع الرئيسي) $ git switch main # 4. دمج الفرع التجريبي بالفرع الرئيسي (يتم تشغيله وأنت داخل الفرع الرئيسي main) $ git merge feature-swap-function # 5. حذف الفرع المحلي والتخلص منه بعد الانتهاء من الدمج بنجاح $ git branch -D feature-swap-function

مفهوم طلبات السحب (Pull Requests)

طلب السحب هو آلية مؤسسية تتيح لك رفع التعديلات من فرعك الجانبي أو مستودعك المستنسخ (Fork) إلى المستودع الأصلي. يتيح ذلك لمالك المشروع فحص الأكواد، إجراء المراجعات الأمنية، والنقاش حول البنية الهندسية قبل الضغط على زر الدمج الرسمي (Merge) أو الرفض.

6. التزامن السحابي والمحاذاة الصارمة (git push & git pull)

أوامر التخاطب المباشر مع الخوادم لرفع أو سحب الملفات ومحاذاة الفروع السحابية لضمان تطابق البيئة المحلية مع خوادم GitHub.

zsh - remote_sync
# 1. ربط مستودعك المحلي بالمستودع السحابي الرسمي $ git remote add origin https://github.com/aymanelmasryael/AEL-Sovereign-CS50x-2026-2027.git # 2. فحص روابط الاتصال السحابي الحالية $ git remote -v # 3. سحب آخر التحديثات من الخادم دون دمجها (للفحص الأمني) $ git fetch origin # 4. سحب التحديثات السحابية ودمجها بالفرع المحلي مباشرة (git pull) $ git pull origin main # 5. دفع الكود المحلي إلى السحاب مع تثبيت مسار الاتصال (git push) $ git push -u origin main # 6. دفع فرع جانبي جديد إلى السحاب لفتح طلب سحب $ git push -u origin feature-swap-function # 7. حذف فرع جانبي من السحاب مباشرة $ git push origin --delete feature-swap-function

7. طوارئ النظام واستعادة النقاء التام (Undoing & git reset)

أوامر الصيانة الطارئة التي استعرضها برايان يو للتراجع عن الأخطاء، التخلص من التجارب الفاشلة، وإعادة ضبط المستودع ليكون ناصع النظافة ومطابقاً للسحاب 100%.

zsh - emergency_reset
# 1. إلغاء تعديلات ملف لم يتم حفظه وإعادته لحالته السابقة $ git restore swap.c # 2. إزالة ملف من منصة التجهيز (Staging) مع الحفاظ على التعديلات بداخله $ git reset swap.c # 3. التراجع الفوري عن آخر توثيق (Commit) محلي والتخلص من تعديلاته نهائياً $ git reset --hard HEAD~1 # 4. التراجع بالزمن إلى توثيق محدد برمز الـ Hash (يكفي وضع أول 6-7 أحرف من معرف التوثيق) $ git reset --hard 4761626 # 5. الأمر الصارم لمطابقة الكود المحلي بالسحاب 100% وإزالة أي أخطاء أو شوائب محلية نهائياً $ git reset --hard origin/main

8. أوامر التحكم السحابي الخارق عبر `gh CLI`

الأسلحة المتقدمة التي تميز كبار مهندسي النظم، وتتيح التحكم بسيرفرات GitHub وطلبات السحب والأتمتة من الطرفية مباشرة.

zsh - gh_cli_elite
# 1. المصادقة وتسجيل الدخول السحابي الآمن إلى حسابك في GitHub $ gh auth login # 2. إنشاء مستودع سحابي جديد ورفع الكود إليه بضغطة زر واحدة $ gh repo create aymanelmasryael/AEL-New-Project --public --source=. --remote=origin --push # 3. السطر الذهبي لضبط إعدادات المستودع (تفعيل النقاشات وفرض الدمج النظيف والحذف التلقائي) $ gh repo edit aymanelmasryael/AEL-Sovereign-CS50x-2026-2027 --delete-branch-on-merge --enable-discussions --enable-squash-merge # 4. فتح طلب سحب رسمي (Pull Request) من موجه الأوامر $ gh pr create --title "🎨 [UI/UX] Showcasing Master Architectural Banner" --body "Injecting master banner for canonical visual authority." # 5. عرض قائمة طلبات السحب المفتوحة حالياً في المستودع $ gh pr list # 6. فحص تفاصيل وحالة طلب سحب محدد برقم الـ ID $ gh pr view 2 # 7. السطر الصارم لدمج طلب السحب أوتوماتيكياً بصلاحيات المالك العليا وحذف الفرع الجانبي $ gh pr merge 2 --squash --delete-branch --admin # 8. استعراض قائمة أحدث عمليات الفحص والبناء الجارية في سيرفرات GitHub Actions $ gh run list --limit 5 # 9. سحب تقرير الحالة الحية لعملية فحص محددة ومعرفة تفاصيل النجاح أو الفشل $ gh run view 28381788686

9. الأسرار التحتية لنظام CS50: كيف يعمل `submit50`؟ (Behind the Scenes)

يختتم المهندس برايان يو ندوة CS50 بالكشف عن السر الهندسي المذهل وراء أداة تسليم الواجبات الشهيرة submit50. يتضح أن الأداة ليست سوى غلاف ذكي (Smart Wrapper) يقوم بتنفيذ أوامر Git و GitHub بالنيابة عن الطالب عبر خط أنابيب أوتوماتيكي متقن:

خط الأنابيب السري لأداة submit50:

  • مستودع لكل طالب: عند تسجيلك في الكورس، يتم إنشاء مستودع خاص بك على خوادم GitHub تحت النطاق github.com/submit50/username.
  • فرع مخصص لكل واجب (Isolated Branching): عندما تكتب في الطرفية أمراً مثل submit50 pset7، تقوم الأداة في الخلفية بإنشاء فرع جديد يحمل اسم الواجب pset7 (أو mario، speller).
  • الدفع السحابي الأوتوماتيكي: تقوم الأداة بعمل git commit لملفاتك الحالية ثم تدفعها مباشرة git push إلى ذلك الفرع المخصص على حسابك في submit50.
  • تجميد الإصدار للتصحيح (Releases & Tagging): بمجرد اكتمال الرفع، تقوم روبوتات تصحيح CS50 بإنشاء "علامة إصدار" (Git Tag / Release) لتجميد هذه النسخة المحددة واعتمادها للتصحيح الأوتوماتيكي وإصدار الدرجة النهائية!
